## 二、8 个分析维度

### 1. 活力节律（Vitality Rhythm）
- **数据源**：`AppState.vitalityData(days30)` — 过去30天各时段（上午/下午/晚上）完成任务数
- **算法**：找最大值时段（`bestBlk`）和最小值时段（`worstBlk`）
- **输出**：效率峰值时段，用于任务分配建议
- **置信度阈值**：数据量 ≥ 20 条 → high，≥ 7 条 → medium，否则不显示

### 2. 完成率偏差（Completion Rate by Block）
- **数据源**：`TaskModel.originalTimeBlock` vs `t.done + t.doneAt`，过去14天
- **算法**：`done_count / planned_count` per block，找完成率最低时段
- **输出**：最低完成率时段及建议（减少任务 or 换时段）
- **置信度阈值**：完成率 < 40% → high，< 60% → medium

### 3. 偏差方向（Systematic Delay/Advance）
- **数据源**：`TaskModel.originalTimeBlock`（规划）vs `doneTimeBlock/doneHour`（实际），过去30天
- **算法**：`avgDev = Σ(actualBlkIdx - plannedBlkIdx) / count`
  - 正数 = 系统性推迟，负数 = 系统性提前
- **输出**：推迟/提前习惯描述及应对建议
- **置信度阈值**：样本量 ≥ 5 才计算；|avgDev| > 0.5 才显示

### 4. 专注深度（Focus Depth）
- **数据源**：`TaskModel.focusSecs`（完成且 > 60s 的任务）
- **算法**：`avgFocusSecs = Σ focusSecs / count`
- **输出**：
  - ≥ 25 分钟：深度工作能力良好
  - 15–25 分钟：中等，给出提升建议
  - < 15 分钟：偏短，给出专注力训练建议
- **置信度阈值**：样本量 ≥ 15 → high，≥ 5 → medium

### 5. 标签负载（Tag Backlog）
- **数据源**：未完成且未忽略任务的 `tags` 字段
- **算法**：`pendingByTag: Map<String, int>`，找 pending 最多的标签
- **输出**：积压最多的标签及集中处理建议
- **阈值**：积压 ≥ 3 件才显示

### 6. 逾期雪球（Overdue Snowball）
- **数据源**：`t.originalDate < today && !t.done && !t.ignored`
- **算法**：`overdueScore = Σ(today - originalDate).inDays`（加权天数，越久越重）
- **输出**：
  - overdueScore > 14：积压严重，给出清理策略
  - 否则：少量积压，给出处理建议
- **置信度**：始终 high（逾期是客观事实）

### 7. 周期规律（Weekly Pattern）
- **数据源**：过去30天完成任务的 `doneAt` → `weekday`
- **算法**：`byWeekday: Map<int, int>`，找最高/最低星期
- **条件**：total30 ≥ 10 且有 ≥ 3 个不同星期几的数据
- **输出**：效率最高/最低的星期，给出任务安排建议

### 8. 近期趋势（7-Day Trend）
- **数据源**：`AppState.doneOnDay(d)` for days7
- **算法**：前3天均值 vs 后3天均值，差值 > 1 判定为上升/下滑
- **输出**：
  - 📈 上升：保持节奏，避免过劳
  - 📉 下滑：排查原因（任务过多/睡眠不足）

## 三、任务量预警（Load Warning）

```dart
estH = todayPending * avgFocusH
if estH > availH * 1.3:  ⚠ 超载警告（建议移除 N 件）
if estH < availH * 0.5:  ✓ 欠载提示（可适当加码）
```

- `availH` = `AppSettings.disposableHours`（用户在设置中填写的可用时间）
- `avgFocusH` = 历史平均专注秒数 / 3600，无数据时默认 25 分钟

## 六、v3 新增维度（β.0.013）

### 9. 命名模式分析（Name Pattern Analysis）
- **数据源**：`TaskModel.text`（所有已完成且有 focusSecs 的任务）
- **算法**：正则 + 数字占比判断
  - **编码式（code）**：名称主体是数字串、字母数字混合，或短码（如 "58 38"、"5839"、"L4-02"）
    - 判断条件：`^[\d\s\-_\.#]+$` 匹配，或数字字符占比 > 40%，或纯英文长度 ≤ 8
  - **自然语言（natural）**：含中文且数字占比 < 20%（如 "背单词"、"过语法"）
  - **混合（mixed）**：其余
- **输出**：各类命名模式的平均专注时长、样本量、代表性例子

### 10. 标签 × 时段完成率交叉（Tag × Block Cross Analysis）
- **数据源**：`TaskModel.tags` × `originalTimeBlock` × `doneAt`，过去30天
- **算法**：按 (tag, block) 分组统计完成率，找每个标签完成率最高的时段
- **输出**：「某标签在某时段完成率 X%」，指导任务分配
- **阈值**：完成率 ≥ 60% 且样本量充足才显示

### 11. 命名模式 × 专注时长交叉（Name × Focus Cross Analysis）
- **数据源**：所有已完成任务的 `text` + `focusSecs`
- **算法**：
  1. 按命名模式分桶（code / natural / mixed）
  2. 计算各桶平均 focusSecs
  3. 若两桶差值 > 10分钟，生成「命名风格影响专注时长」洞察
- **输出示例**：
  > 「数字/编码式」命名的任务比「文字描述」任务平均多专注 23 分钟。编码类工作需要更深度的投入，规划时留足整块时间。

## 七、任务分配建议逻辑升级（v3）

v3 的任务分配建议在原有「逾期/重要→最佳时段」逻辑基础上，新增：

1. **编码式命名推断**：若任务名称被识别为编码式（如 "58 40"），且历史同类任务平均 > 40 分钟，自动建议安排在效率峰值时段
2. **标签×时段匹配**：查找该任务标签在哪个时段历史完成率最高，优先推荐该时段
3. **样本不足时降级**：数据不足 3 条时，退回至默认逻辑（避免错误建议）

*最后更新：β.0.013 / 2026-03*

## 八、v4 命名模式扩展（β.0.014）

v3 只有 code / natural / mixed 3 类，样本分散无统计意义。v4 扩展为 **8 类**：

| 枚举值 | 说明 | 识别规则 |
|--------|------|---------|
| `code` | 纯数字/编码式 | 正则 `^[\d\s\-_\.#]+$`，或数字占比 > 45%，或纯英数 ≤ 6 字符 |
| `physics` | 物理学科 | 命中物理词典（100+ 术语）：力学、量子、电磁、光学、热力学… |
| `medicine` | 临床医学 | 命中医学词典（120+ 术语）：内科、外科、病理、诊断、药理… |
| `lang_study` | 语言学习 | 命中语言学习词典（80+ 术语）：单词、语法、雅思、托福、听力… |
| `daily_cn` | 中文日常 | 命中日常生活词典（50+ 词）：打扫、购物、联系、运动… |
| `task_verb` | 动词驱动任务 | 任务名以动词前缀开头：写、做、复习、整理、提交、检查… |
| `mixed` | 混合式 | 中英文混排，或含数字 10-45% |
| `unknown` | 其他 | 无法匹配任何规则 |

### 深度工作类型判断
`code`、`physics`、`medicine`、`lang_study` 被识别为"深度工作"类型，若历史均专注时长 > 35 分钟，自动推荐安排在效率峰值时段。

### 交叉洞察逻辑
- 找出专注时间最长和最短的两个命名模式，若差值 ≥ 8 分钟，生成对比洞察
- 对每个有足够样本（≥ 3 件）的命名模式，按均时长分级给出建议：
  - ≥ 45 分钟：深度工作，整块时间
  - 25-45 分钟：标准番茄钟
  - < 25 分钟：碎片化批处理
